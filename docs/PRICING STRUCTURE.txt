FLYN AI — Billing & Entitlement Architecture
1️⃣ Core Design Principles (Non-Negotiable)
1. Entitlements ≠ Billing

○ Billing answers “who pays and how much”
○ Entitlements answer “what can this tenant do right now”

2. Every request checks entitlements

○ UI visibility is cosmetic
○ Backend enforcement is mandatory

3. Usage is metered independently

○ Billing reads usage
○ Entitlements gate execution

Tenants
tenants

- plan_id
- status (trial | active | past_due | suspended)
- region
- created_at

Plans
plans
- id
- name (FREE, STARTER, GROWTH, PRO, ENTERPRISE)
- billing_cycle (monthly | yearly)
- is_public

Plan Entitlements
plan_entitlements

- feature_key
- value (boolean | integer | string)

Atomic Usage Increment (MANDATORY)
Every cost-driving action must:
1. Check entitlement
2. Check usage limit
3. Increment atomically
4. Execute action

if !entitled("channels.whatsapp"):
  deny()

  throttle()

increment_usage("messages.sent", +1)
send_message()

5️⃣ Entitlement Resolution Logic (How the System Thinks)
Resolution Order
1. Tenant override (highest priority)
2. Plan entitlement
3. Default = false

    if tenant_override exists:
        return override.value
    if plan_entitlement exists:
        return plan.value
    return false

5️⃣ Atomic Metering Flow (MANDATORY)
Every cost-driving action follows this sequence:
1. Check tenant status == active
2. Check entitlement(feature)
3. Check usage < limit
4. Reserve usage (atomic increment)
5. Execute action
6. Finalize usage (if needed)

  deny

  throttle
increment(messages.sent, +1)
send_message()


6️⃣ Voice Metering (Special Handling)
Call Lifecycle
1. Call starts → provisional session
2. Call ends → duration calculated
3. Minutes rounded UP
4. Increment calls.minutes
minutes = ceil(duration_seconds / 60)
increment(calls.minutes, minutes)

If limit exceeded mid-call:
● Allow call to finish
● Block next call
● Prompt upgrade
